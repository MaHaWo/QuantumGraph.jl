QuantumGravPy Repository Overview

Python package for graph neural networks used in causal-set/quantum-gravity experiments. The central flow is: YAML config with custom tags → zarr causal-set data → PyTorch Geometric Data objects → configurable GNN model → training/evaluation/checkpointing → optional Optuna tuning.

Tech stack

  • Language/package: Python 3.12+, hatchling build backend, src-layout package QuantumGrav.
  • ML/GNN: PyTorch, PyTorch Geometric, torch.distributed/DDP.
  • Data: zarr LocalStore, NumPy, pandas, PyYAML, joblib parallelism.
  • Config/validation: JSON Schema via jsonschema, custom PyYAML tags !pyobject, !sweep, !coupled-sweep, !range, !random_uniform, !reference.
  • Tuning: Optuna with JournalStorage/RDBStorage and MedianPruner.
  • Testing/docs: pytest, pytest-cov, mkdocs-material, mkdocstrings, mkdocs-jupyter.

Dominant paradigms: object-oriented configurable components for models/trainers/evaluators; PyTorch module composition for numerics; YAML-driven dependency injection; unit-test-heavy pytest coverage using synthetic zarr fixtures.

Repository structure

src/QuantumGrav/

Package root and public API. Exports dataset, GNN model, trainers, evaluators, early stopping, config helpers, zarr loaders, and models subpackage.

src/QuantumGrav/models/

Reusable neural network blocks: GNNBlock, LinearSequential, SkipConnection.

src/QuantumGrav/QGTune/

Optuna search-space construction, reference resolution, study creation, and best-config export.

test/

Unit tests and synthetic fixtures for config, datasets, models, training/DDP, evaluation, tuning, and utility helpers.

docs/

MkDocs user documentation: getting started, datasets, models, training, tuning, API.

Key modules

  • __init__.py: public import surface; another package can import QuantumGrav as QG and access trainers, datasets, evaluators, config utilities, zarr loaders, and models.
  • base.py: abstract Configurable protocol for from_config/to_config.
  • config_utils.py: YAML constructors, inclusive ranges, Python-object loading, sweep expansion with coupled sweep lockstep, and conversion of non-builtin values back to !pyobject tags.
  • dataset_base.py + dataset_ondisk.py: zarr-backed dataset layer. Counts samples via num_causal_sets/num_samples, scalar-array fallback, and a Julia-zarr fallback on adjacency_matrix. Can preprocess chunks to processed/data_*.pt or lazily read per item.
  • gnn_model.py: dynamic model composition. Instantiates encoders, pooling/latent pathway, graph-features pathway, and multiple downstream task heads from classes/instances/callables.
  • models/: GNNBlock combines PyG convolution, normalization, activation, skip, dropout; LinearSequential builds linear/activation stacks; SkipConnection projects residuals when feature dims differ.
  • evaluate.py: base evaluation loop collecting losses, predictions, targets and optional monitor metrics into a pandas DataFrame. Validator and Tester specialize reporting.
  • early_stopping.py: metric-oriented early stopping across one or more tasks using min/max modes, deltas, grace periods, patience, and any/all aggregation.
  • train.py: main training orchestrator: validates config, constructs model/optimizer/scheduler/dataset/loaders, runs epochs, validation, Optuna pruning, checkpointing, and testing.
  • train_ddp.py: DDP extension with process-group helpers, per-rank device setup, DDP model wrapping, and DistributedSamplers.
  • QGTune/tune.py: turns custom YAML sweep/range/random nodes into Optuna suggestions, resolves references/coupled sweeps, creates studies, and writes best configs.
  • load_zarr.py: utility to recursively convert zarr groups to nested dictionaries.

Entry points and reading order

  1. Start at src/QuantumGrav/__init__.py to see the public API and package boundaries.
  2. For the main runtime path, read src/QuantumGrav/train.py around Trainer.__init__, prepare_dataloaders, run_training, and run_test.
  3. Then inspect src/QuantumGrav/gnn_model.py to understand model composition and forward data shape expectations.
  4. For data migration or Julia interoperability, prioritize dataset_base.py, dataset_ondisk.py, test/conftest.py, and docs docs/datasets_and_preprocessing.md.
  5. For hyperparameter/config behavior, read config_utils.py and QGTune/tune.py.

Tests and fixtures

Tests are pytest unit tests under test/. They rely heavily on fixtures in test/conftest.py, which create temporary zarr stores with arrays adjacency_matrix, link_matrix, max_pathlen_future, max_pathlen_past, dimension, atomcount, and sometimes num_samples. Reader fixtures convert dense adjacency to PyG sparse edge_index/edge_attr and build node features from path-length arrays.

  • Config: test_config_utils.py, test/config.yaml cover YAML custom tags and sweep expansion.
  • Data/zarr: test_datasetbase.py, test_ondiskdataset.py, test_load_zarr.py cover zarr reading, sample counting, preprocessing, indexing, dataloaders, recursive zarr-to-dict.
  • Model blocks: test_gnn_block.py, test_skipconnection.py, test_linearsequential.py, test_gnn_model.py cover initialization, forward/backward, config, save/load, graph features, latent/pooling paths, arbitrary composition, active tasks.
  • Training/evaluation: test_trainer.py, test_trainer_ddp.py, test_evaluate.py, test_early_stopping.py cover trainer lifecycle, dataloaders, epoch loops, checkpointing, DDP setup, evaluators, and early-stopping behavior.
  • Tuning/utilities: test_tune.py, test_utils.py cover Optuna suggestions, references, study storage, best-config save, nested path helpers, dynamic imports.

Constraints, risks, and open questions for migration to Julia

  • PyTorch/PyG coupling: core models and datasets depend on torch.nn.Module, torch_geometric.data.Data, PyG layers/loaders, and PyTorch checkpoint formats. A Julia port must choose equivalents (e.g. Flux/GraphNeuralNetworks.jl) or maintain an interop boundary.
  • Dynamic Python object injection: configs carry class/function objects via !pyobject. Julia migration needs a registry or explicit mapping rather than arbitrary import strings.
  • Data format ambiguity: zarr sample counting has multiple fallbacks and explicitly handles Python-vs-Julia zarr layout differences. Preserve this behavior and add golden zarr fixtures before changing readers.
  • Model save/load: GNNModel saves only state_dict and requires matching config; LinearSequential saves config + state_dict. Cross-language migration must define a stable serialization contract.
  • Evaluator/loss signatures: trainer criterion expects criterion(outputs, data, trainer) in training, while evaluator criterion uses criterion(outputs, data). This mismatch is an important API quirk.
  • DDP assumptions: DDP defaults to NCCL and one process per GPU; CPU tests may not cover cluster behavior thoroughly.
  • Mutable config/state: ConfigHandler mutates copied configs and appends run names; early stopping mutates task dictionaries by adding runtime fields. Be careful with aliasing when porting.
  • Tests use random synthetic data: fixtures use random data without explicit seeding in places; exact numeric behavior is not a stable oracle.
